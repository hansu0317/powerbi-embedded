1️⃣ 프로젝트 개요

Microsoft Fabric(app.powerbi.com)에 게시된 Power BI 보고서를 Power BI 라이선스가 없는 직원도 볼 수 있게 해주는 사내 게이트웨이 서버입니다.
직원은 브라우저로 사내 서버에 접속해 로그인하면, 본인에게 허용된 보고서만 확인할 수 있습니다.

추가 비용없음 (기존 Power BI 라이선스 활용)

3️⃣ 전체 동작 흐름 (단계별)
브라우저에서 서버 접속
직원이 http://192.168.56.10:8082 접속. 서버가 세션 쿠키 확인 → 로그인 안 된 상태면 /login으로 자동 이동(302 리디렉션).
로그인 (아이디 + 비밀번호)
POST /login으로 폼 데이터 전송 → 서버가 PostgreSQL users 테이블에서 username 조회 → bcrypt.checkpw()로 비밀번호 검증 → 성공 시 세션 쿠키 발급(8시간 유효) + /로 이동. 실패 시 에러 메시지 표시. 모든 시도는 logs/server.log에 기록.
보고서 목록 표시 (사용자별 필터)
GET / 요청 → 세션에서 username 추출 → PostgreSQL user_reports 테이블 조회 → 이 사용자에게 허용된 보고서만 왼쪽 사이드바에 표시. 다른 보고서는 존재 자체를 모릅니다 (HTML에 없음).
보고서 클릭 → AJAX 요청
사이드바에서 보고서 클릭 → 브라우저가 GET /api/embed/{보고서명} AJAX 요청 → 서버가 ① 로그인 여부 확인 ② 해당 보고서 권한 확인 → 권한 없으면 403 에러 반환(페이지 이동 없음).
Azure AD 토큰 발급 (서버 ↔ Azure)
서버가 msal로 Azure AD에 Client Credentials 토큰 요청 (CLIENT_ID + CLIENT_SECRET 사용, 사람이 개입하지 않는 자동 인증) → 유효한 토큰이 캐시에 있으면 재사용(1시간, 만료 5분 전 자동 갱신).
Power BI Embed Token 발급 (서버 ↔ Fabric)
Azure AD 토큰으로 Power BI API 2번 호출:
① GET .../reports/{reportId} → embedUrl, datasetId 획득
② POST .../GenerateToken → 일회성 Embed Token 발급 (identity: pbi_username + roles 포함 → RLS 적용 기준)
결과를 JSON으로 브라우저에 반환: { embed_token, embed_url, report_id }
보고서 렌더링 (브라우저 ↔ Fabric)
브라우저의 powerbi-client JS SDK가 Embed Token + embedUrl로 Microsoft Fabric 서버에 직접 접속 → 보고서 렌더링. 이 단계부터는 서버가 개입하지 않습니다. 데이터는 브라우저 ↔ Fabric 직접 통신.
탭 관리 (SPA)
다른 보고서 클릭 → 4~7단계 반복, 새 탭으로 추가. 탭 전환 시 이미 로드된 보고서는 숨기기/보이기만 해서 재로드 없이 즉시 전환. 탭 닫기 시 pbi.reset()으로 SDK 인스턴스 정리 후 DOM 제거.

6️⃣ API 엔드포인트
메서드경로설명인증

왜 GET을 쓰는가? (POST 대신)

/api/embed/{name}은 GET으로 구현되어 있습니다. 이유:

• 세션 쿠키가 없으면 서버가 401 반환 — URL만 알아도 토큰 발급 불가
• 권한 없는 보고서는 서버가 403 반환 — 세션이 있어도 본인 것만 접근
• 사내 내부망 전용 서버라 로그에 URL 경로가 남아도 외부 유출 위험 없음

POST로 바꾸면 JS 코드(fetch 호출 방식)도 함께 수정해야 하는 번거로움이 있고, 보안상 실질적인 개선이 없어 GET을 유지합니다.

레이어 1 — 게이트웨이 접근 제어 ✅ 적용

누가 어떤 보고서 탭을 볼 수 있는지 제어

• PostgreSQL user_reports 테이블 기반
• 권한 없는 보고서는 사이드바에 아예 안 보임
• AJAX 요청 시에도 서버에서 403 반환
• 현재 완전히 작동 중

레이어 2 — Fabric 데이터 RLS ⚠️ Fabric 설정 필요

보고서 안에서 어떤 데이터 행이 보이는지 제어

• GenerateToken에 pbi_username + roles 전달 → 메커니즘은 구현됨
• 현재 "도메인" 역할 = DAX 필터 없음 = 전체 데이터 표시
• Fabric에서 HR_RLS 역할에 DAX 필터 설정 시 즉시 작동

아키텍처 — PostgreSQL이 RLS에서 맡는 역할

두 레이어 모두 출발점은 PostgreSQL이다. 레이어 1은 user_reports가, 레이어 2는 users.pbi_username + users.roles 두 컬럼이 결정한다. 즉 RLS 적용/변경은 Fabric의 DAX 필터 설정 + PostgreSQL UPDATE 한 쌍의 작업이다.
[PostgreSQL powerbi_gateway] [FastAPI 게이트웨이] [Microsoft Fabric] users.username / password ────▶ ① 로그인 인증 (bcrypt 비교) user_reports ────▶ ② 보고서 접근 권한 체크 ← 레이어 1 (탭 단위 차단) report_meta.pbi_report_id ────▶ ③ DB에서 Fabric Report ID 조회 users.pbi_username ┐ users.roles ┴───────▶ ④ GenerateToken identity 전달 ──▶ ⑤ roles와 같은 이름의 RLS 역할 탐색 { username: pbi_username, DAX 필터의 USERNAME() 자리에 roles: [roles] } pbi_username 값이 대입됨 ← 레이어 2 (행 단위 필터)

RLS 적용 시 PostgreSQL 쪽 작업 목록

순서SQL 작업설명
1 UPDATE users SET roles = 'RLS_사번' WHERE username = '...'; Fabric에 만든 역할명과 철자까지 일치해야 함 (콤마로 여러 역할 가능)
2 UPDATE users SET pbi_username = '1001' WHERE username = '...'; DAX 필터의 USERNAME()이 받게 될 값 — 사번이든 이메일이든 DAX가 비교하는 컬럼 기준으로
3 user_reports는 그대로 유지 보고서 접근 권한(레이어 1)은 RLS와 독립 — RLS를 켜도 탭 권한 관리는 기존 방식 그대로
순서 주의: DB의 roles를 먼저 바꾸면 안 된다. Fabric 데이터셋에 해당 역할이 아직 없는 상태에서 로그인하면 MSOLAP 연결 오류로 보고서가 아예 안 열린다. 반드시 Fabric DAX 역할 생성 → 게시 → DB UPDATE 순서로.

레이어 2가 필요한가? — 판단 기준

상황결론
같은 보고서를 볼 수 있는 사람들이 모두 동일한 데이터를 봐도 된다 레이어 1만으로 충분 — 현재 완성
같은 보고서인데 사람마다 보이는 데이터 행이 달라야 한다
(예: 팀장은 전체 공장, 직원은 본인 공장만)		 레이어 2 필요 — Fabric DAX 설정 + DB roles 변경
코드는 이미 준비되어 있습니다. GenerateToken 호출 시 pbi_username과 roles를 전달하는 구조가 구현되어 있어, Fabric에서 해당 역할에 DAX 필터만 설정하면 즉시 작동합니다.

레이어 2 활성화 방법 (Fabric 담당자 필요)

// Fabric에서 HR_RLS 역할 DAX 필터 예시 [사번] = USERNAME() // 또는 부서 기반 [부서코드] = LOOKUPVALUE(직원[부서], 직원[이메일], USERNAME()) -- DAX 설정 후 DB에서 역할 변경 UPDATE users SET roles = 'HR_RLS', pbi_username = '실제AD이메일@qualisoft.co.kr' WHERE username = 'user01';

쉽게 이해하기 — RLS는 "행 거름망"이다

RLS는 보고서를 숨기는 기능이 아니다. 같은 보고서를 열어도 안에 있는 데이터 표의 행(row)이 사람마다 다르게 걸러져 보이는 것이다.

예를 들어 생산현황 보고서 안에 이런 데이터가 있다고 하면:
사번이름생산량
1001김철수500
1002이영희300
1003박민수700
계정roles (DB)pbi_username (DB)같은 보고서를 열면
user01 도메인 (필터 없음) user01@qualisoft.co.kr 3행 전부 보임 (현재와 동일)
rls01 (RLS 데모 계정) RLS_사번 1001 ← 사번을 넣음 1001 김철수 행만 보임, 차트도 500만 집계

동작 원리 — 양쪽이 한 쌍으로 맞아야 한다

[사내 서버] rls01 로그인 │ GenerateToken 호출: { username: "1001", roles: ["RLS_사번"] } ▼ [Fabric] 데이터셋에서 "RLS_사번" 역할을 찾음 │ 역할의 DAX 필터: [사번] = USERNAME() │ → USERNAME() 자리에 서버가 보낸 "1001"이 들어감 ▼ [결과] 사번이 1001인 행만 브라우저로 전달
핵심: pbi_username은 실제 이메일일 필요가 없다. DAX의 USERNAME()이 돌려받는 그냥 문자열이므로, Fabric 필터가 [사번] = USERNAME()이면 사번을, [이메일] = USERNAME()이면 이메일을 넣으면 된다. "Fabric의 DAX 필터가 무엇과 비교하는가"에 맞춰 DB 값을 넣는 것이 전부다.

자주 보는 오류 — "Failed to open the MSOLAP connection"

DB에는 roles = 'RLS_사번'으로 넣었는데 Fabric 데이터셋에 그 이름의 역할이 아직 없으면 Fabric이 모델 연결 자체를 거부하면서 이 오류가 난다.

• 서버는 "RLS_사번 역할로 봐주세요"라고 요청만 하는 입장
• 역할의 실체(DAX 필터)는 Power BI Desktop에서 만들어 게시해야 Fabric에 생긴다
• 같은 보고서를 도메인 역할 계정(user01 등)으로 열면 정상 → 역할 부재가 원인이라는 증거

로그 롤링 (Log Rotation)

서버를 시작할 때마다 기존 logs/server.log를 어제 날짜 폴더로 자동 이동하고 새로 시작합니다.
매일 아침 출근 후 서버를 켜면 전날 로그가 날짜 폴더에 보관됩니다.

logs/20260609/server.log ← 어제 로그
logs/server.log ← 오늘 현재 로그

server.sh 동작 원리

#!/bin/bash PYTHON=/home/test01/.pyenv/versions/3.11.9/bin/python3 PID_FILE="$PROJECT_ROOT/.server.pid" LOG_FILE="$LOG_DIR/server.log" # ── rotate_log(): 기존 로그 → 어제 날짜 폴더로 이동 rotate_log() { YESTERDAY=$(date -d "yesterday" +%Y%m%d) # 20260609 mv "$LOG_FILE" "$LOG_DIR/$YESTERDAY/server.log" } # start() 진입 시 rotate_log() 호출 → 매일 아침 start하면 전날 로그 보관 # ── start(): 프로세스 시작 start() { # kill -0 PID: 프로세스 존재 여부만 확인 (실제 신호 보내지 않음) if [ -f "$PID_FILE" ] && kill -0 "$(cat $PID_FILE)" 2>/dev/null; then echo "이미 실행 중"; return fi rotate_log # nohup: 터미널 닫혀도 프로세스 유지 # &: 백그라운드 실행 # $!: 방금 실행한 프로세스 PID → PID_FILE에 저장 nohup $PYTHON -m uvicorn main:app --host 0.0.0.0 --port 8082 > "$LOG_FILE" 2>&1 & echo $! > "$PID_FILE" } # ── stop(): 프로세스 종료 stop() { PID=$(cat "$PID_FILE") kill "$PID" # SIGTERM → uvicorn 정상 종료 rm -f "$PID_FILE" # PID 파일 제거 } # restart = stop → sleep 1 → start
PID 파일(.server.pid)이란?
실행 중인 서버 프로세스 ID를 저장하는 텍스트 파일. stop이나 status 시 이 파일을 읽어 어떤 프로세스를 제어할지 알 수 있습니다. 비정상 종료로 PID 파일이 남아있을 경우 rm -f .server.pid로 수동 삭제 후 start 가능.
